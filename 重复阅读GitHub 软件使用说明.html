GitHub 软件使用说明

把 GitHub 当成“带版本记录的项目资料库 + 团队协作平台”。你可以用它保存代码、文档、表格、网页文件，也可以用它记录问题、审核修改、发布简单网站。

1. GitHub 是什么？

一句话理解：GitHub 是一个在线平台，用来保存项目文件、记录每次修改、多人协作讨论，并且可以发布开源项目或内部项目。

资料库

像网盘

可以放代码、说明书、图片、网页文件、配置文件等。

版本记录

比网盘强

每次修改都会留下记录，知道谁改了、改了什么、什么时候改的。

团队协作

像项目管理工具

可以提问题、分配任务、审核修改、合并成果。

重要提醒：GitHub 不等于普通网盘。公开仓库 Public 里的内容别人可以看到；公司内部资料、账号密码、客户信息不要放到公开仓库。

Git 和 GitHub 的区别

名称 你可以怎么理解 用途
Git 版本控制工具 记录文件修改历史，支持回退、分支、合并。
GitHub 云端平台 把 Git 项目放到网上，方便备份、协作、审核、发布。

2. 必懂术语：先背这 10 个

你不需要一开始就会写代码，但这些词必须先看懂。

术语 中文理解 通俗解释
Repository / Repo 仓库 一个项目的文件夹。例如：一个网站、一个软件、一个说明文档项目。
README.md 项目说明书 仓库首页默认展示的说明文件，通常写项目介绍、使用步骤、注意事项。
Commit 提交 保存一次修改记录。类似“保存一个版本快照”。
Branch 分支 从主线复制出一条修改线，先在分支上改，确认没问题再合并。
Pull Request / PR 合并请求 告诉团队：“我改好了，请检查，没问题就合并到主线。”
Issue 问题 / 任务 用来记录 Bug、需求、待办、反馈。
Clone 克隆 把 GitHub 上的仓库完整下载到自己电脑。
Push 推送 把自己电脑上的修改上传到 GitHub。
Pull 拉取 把 GitHub 上别人更新的内容同步到自己电脑。
Fork 复制别人的仓库 把别人项目复制一份到自己账号里，方便学习或修改。

3. 30 分钟快速入门：按这个做一遍

目标：创建一个仓库，上传一个文件，修改说明书，学会查看版本记录。

1

注册 GitHub 账号

  1. 打开 GitHub 官网。
  2. 点击右上角 Sign up。
  3. 填写邮箱、密码、用户名。
  4. 进入邮箱完成验证。
用户名建议简短、专业、容易记。公司项目建议用公司统一邮箱注册或加入组织。
2

创建你的第一个仓库

  1. 登录后，点击右上角 +。
  2. 选择 New repository。
  3. Repository name 填：my-first-github-project。
  4. Description 填：我的第一个 GitHub 项目。
  5. 选择 Private，先用私有仓库练习。
  6. 勾选 Add a README file。
  7. 点击 Create repository。
3

编辑 README.md

  1. 进入仓库首页，点击 README.md。
  2. 点击右上角铅笔图标 Edit this file。
  3. 输入下面内容：
# 我的第一个 GitHub 项目

这是我学习 GitHub 的练习仓库。

## 我学会了什么
- 创建仓库
- 编辑 README
- 提交 Commit
- 查看修改历史
  1. 页面下方 Commit message 填：更新 README 说明。
  2. 点击 Commit changes。
4

上传一个文件

  1. 回到仓库 Code 页面。
  2. 点击 Add file。
  3. 选择 Upload files。
  4. 拖入一个测试文件，例如图片、TXT、HTML。
  5. Commit message 填：上传测试文件。
  6. 点击 Commit changes。
5

查看版本记录

  1. 仓库首页点击右侧或文件列表上方的 Commits。
  2. 你会看到每一次保存记录。
  3. 点开某一次 Commit，可以看到具体改了哪些内容。
完成标准：你已经会创建仓库、编辑文件、上传文件、提交版本、查看历史。

4. 仓库 Repository：项目的“总文件夹”

一个仓库就是一个项目。建议一个项目一个仓库，不要把所有资料都塞进一个仓库。

创建仓库时怎么选？

选项 建议 说明
Repository name 用英文、小写、短横线 例如：amazon-sofa-listing、product-research-template。
Public 开源项目才选 别人可以看到，不适合放公司资料。
Private 练习和公司资料优先选 只有你和被邀请的人能看。
Add README 建议勾选 方便直接在首页写说明。
.gitignore 新手可先不选 用于忽略不需要上传的文件。
License 公开项目建议选 说明别人能不能使用你的代码或内容。

推荐仓库结构

my-project/
├─ README.md              # 项目说明
├─ docs/                  # 文档资料
├─ images/                # 图片
├─ src/                   # 代码文件
├─ data/                  # 数据文件
└─ notes/                 # 会议记录、想法
命名建议：文件名尽量用英文，例如 product-research.md，少用空格和特殊符号。这样后续协作、下载、部署网页更稳定。

5. 上传、编辑、删除文件

上传文件

  1. 进入仓库首页。
  2. 点击 Add file。
  3. 选择 Upload files。
  4. 拖拽文件到页面。
  5. 填写 Commit message。
  6. 点击 Commit changes。

新建文件

  1. 进入仓库首页。
  2. 点击 Add file。
  3. 选择 Create new file。
  4. 输入文件名，例如 notes/meeting.md。
  5. 填写内容并提交。

如何创建文件夹？

GitHub 不是单独点击“新建文件夹”，而是在新建文件时，通过文件名带斜杠来创建。

docs/how-to-use.md

这表示：创建 docs 文件夹，并在里面创建 how-to-use.md 文件。

删除文件

  1. 点开要删除的文件。
  2. 点击右上角 ... 或垃圾桶图标。
  3. 填写删除原因，例如：删除无用测试文件。
  4. 点击 Commit changes。
注意：删除文件不是彻底消失，历史版本里仍可能看到旧内容。不要上传密码、密钥、客户资料等敏感信息。

6. 版本记录 Commit：GitHub 最重要的能力

Commit 可以理解为“保存一个版本快照”。每次重要修改都提交一次。

好的 Commit message 怎么写？

不推荐 推荐 原因
update 更新首页产品介绍 别人一看就知道改了什么。
fix 修复按钮无法点击的问题 明确说明修复对象。
111 新增安装步骤说明 方便以后查找历史。

查看某个文件的历史

  1. 点开某个文件。
  2. 点击右上或上方的 History。
  3. 选择某次记录，查看前后变化。

什么时候提交一次？

  • 完成一个小功能时提交。
  • 修改一段文案时提交。
  • 修复一个问题时提交。
  • 不要一天只提交一次超大改动，这样不好检查。

7. 团队协作：Issue / Branch / Pull Request

团队使用 GitHub 的核心流程是：先记录任务，再开分支修改，最后用 Pull Request 审核合并。

Issue 提任务→ Branch 开分支→ 修改文件→ Pull Request 审核→ Merge 合并

7.1 Issue：用来记录问题和任务

Issue 适合记录：Bug、需求、待办事项、用户反馈、会议后要做的事。

  1. 进入仓库。
  2. 点击 Issues。
  3. 点击 New issue。
  4. 标题写清楚，例如：首页图片加载太慢。
  5. 正文写背景、问题、期望结果、截图。
  6. 可以添加 Assignee 负责人、Labels 标签、Milestone 阶段。
  7. 点击 Submit new issue。

Issue 模板

标题：简短说明问题

背景：
为什么要做这件事？

现状：
现在发生了什么？

期望：
希望变成什么样？

附件：
截图、链接、文件

负责人：
@某个人

7.2 Branch：分支，避免直接改主线

主分支通常叫 main。不要随便直接改 main。建议每个任务开一个新分支。

分支名 适合场景
fix-homepage-image 修复首页图片问题
add-product-guide 新增产品说明
update-readme 更新说明文档

7.3 Pull Request：提交给别人检查

Pull Request 简称 PR。它不是“下载请求”，而是“请求把我的修改合并到主线”。

  1. 在分支上完成修改。
  2. 点击 Compare & pull request。
  3. 标题写清楚：修复首页图片加载问题。
  4. 正文说明改了什么、为什么改、如何测试。
  5. 邀请同事 Review。
  6. 确认没问题后点击 Merge pull request。

PR 模板

本次修改：
- 

修改原因：
- 

测试方式：
- 

风险点：
- 

关联 Issue：
Closes #编号
团队最佳实践：主分支只放确认无误的内容；所有较重要修改都通过分支和 PR 完成。

8. GitHub Pages：发布一个简单网页

GitHub Pages 可以把仓库里的网页文件发布成一个可访问的网站，适合放说明页、作品集、项目介绍、简单静态网页。

方法 A：发布个人主页

  1. 点击右上角 + → New repository。
  2. 仓库名必须是：你的用户名.github.io。
  3. 勾选 Add README。
  4. 创建仓库。
  5. 进入 Settings → Pages。
  6. Source 选择 Deploy from a branch。
  7. Branch 选择 main，目录选择 /root。
  8. 保存后访问：https://你的用户名.github.io。

方法 B：发布普通项目网页

  1. 在仓库根目录创建 index.html。
  2. 进入 Settings → Pages。
  3. Source 选择 Deploy from a branch。
  4. Branch 选择 main，目录选择 /root。
  5. 保存，等待发布完成。
注意：GitHub Pages 是静态网站，适合 HTML、CSS、JS。不适合直接运行需要服务器、数据库、后台接口的复杂系统。

最简单的 index.html 示例

<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <title>我的第一个网站</title>
</head>
<body>
  <h1>你好，GitHub Pages！</h1>
  <p>这是我发布的第一个网页。</p>
</body>
</html>

9. GitHub Desktop：不用命令行也能同步

如果你害怕命令行，可以先用 GitHub Desktop。它是图形界面工具，适合新手把仓库下载到电脑、提交修改、推送到 GitHub。

基本流程

  1. 下载安装 GitHub Desktop。
  2. 登录 GitHub 账号。
  3. 选择 Clone a repository from the Internet。
  4. 选择你的仓库，设置本地保存位置。
  5. 在电脑文件夹里修改文件。
  6. 回到 GitHub Desktop，左侧会显示修改内容。
  7. 左下角填写 Summary，例如：更新产品说明。
  8. 点击 Commit to main 或当前分支。
  9. 点击 Push origin 上传到 GitHub。

GitHub Desktop 常用按钮

按钮 含义
Fetch origin 检查 GitHub 上有没有新变化。
Pull origin 把 GitHub 上的新变化下载到电脑。
Push origin 把电脑上的提交上传到 GitHub。
Commit 保存一次本地版本记录。
Current branch 查看或切换当前分支。

10. 命令行进阶：会这几个就够了

刚开始可以不用命令行。等你熟悉 GitHub 后，再学习下面这些。

第一次下载仓库

git clone 仓库地址

查看当前修改

git status

保存一次修改

git add .
git commit -m "这里写本次修改说明"

上传到 GitHub

git push

同步别人更新

git pull

完整日常命令流程

# 1. 先同步最新内容
git pull

# 2. 修改文件后，查看状态
git status

# 3. 添加所有修改
git add .

# 4. 提交版本记录
git commit -m "更新说明文档"

# 5. 上传到 GitHub
git push
新手建议：不要一上来就强行使用 git reset、git rebase、git push --force，这些命令可能覆盖或改乱历史记录。

个人使用 SOP

团队协作 SOP

适合你公司的简单协作规则

场景 建议规则

12. 常见错误和解决办法

问题 原因 解决办法
别人看到了我的文件 仓库设置成 Public 进入 Settings，把仓库改为 Private；敏感信息仍建议更换。
找不到上传按钮 不在仓库 Code 页面 进入仓库首页，点击 Code，再找 Add file。
README 没显示 文件名不正确 必须叫 README.md，通常放在仓库根目录。
Pages 网站没更新 发布需要等待，或设置错误 检查 Settings → Pages 的分支和目录；等待几分钟再刷新。
PR 不能合并 有冲突、检查失败、缺少审核 看 PR 页面提示，解决冲突或等待审核通过。
命令行 push 失败 权限、网络、远程地址或本地落后 先执行 git pull，确认登录和仓库权限。

13. 练习任务：按顺序完成

完成这些练习后，你就具备 GitHub 基础使用能力。

练习 1：创建仓库

  • 创建 Private 仓库。
  • 勾选 README。
  • 写清楚仓库用途。

练习 2：上传文件

  • 上传一张图片。
  • 上传一个 HTML 文件。
  • 提交 Commit。

练习 3：创建 Issue

  • 创建一个任务 Issue。
  • 添加标题、描述、负责人。
  • 添加 Label。

练习 4：创建 PR

  • 新建分支。
  • 修改 README。
  • 提交 Pull Request。

最终你要达到的能力

你能独立完成：创建仓库、上传资料、编辑说明、查看历史、创建任务、开分支、提交 PR、发布简单网页。
